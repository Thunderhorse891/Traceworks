        Phase One Launch Rules

        Paid checkout stays off until the launch audit has zero blockers TraceWorks now refuses new paid orders when live payments, durable storage, secure status tracking, source coverage, or delivery infrastructure are not truly ready.
        Launch claims must be data-driven, not hardcoded This page avoids pretending a key, inbox, or source is configured unless the live environment or admin audit endpoint confirms it.
        Sold package scope must match actual source coverage If a workflow promises appraisal, recorder, mortgage, probate, or licensed people-association coverage, the required source modules must be configured before launch.
        Durable storage is non-negotiable Production must run on REST KV for orders, queue state, dedupe, and report artifacts. Local file storage remains development-only.

        Live Proof Still Required Today

        Real end-to-end launch rehearsal Run one real order on the deployed domain from intake through Stripe, webhook, queue, report generation, artifact delivery, and customer status access.
        County and provider validation Validate the highest-value counties and licensed sources with real identifiers so the report narrative matches the evidence actually returned in production.
        Inbox and artifact delivery proof Confirm payment confirmation, final delivery emails, and authenticated artifact downloads all work from the real deployed environment.
        Queue recovery and operator visibility Watch Netlify function logs, queue depth, retry timing, and dead-letter behavior after a live order so operations are proven, not assumed.
